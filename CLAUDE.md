# Circuit Defense

A browser tower defense game. **The game is `tower-defense.html` (markup) + `tower-defense.css` (styles) + seven domain-split JS files (v1.8.2).** Everything is wired with classic `<link rel="stylesheet">` and `<script src>` tags — **NEVER ES modules** (`type="module"` breaks `file://`). No build step, no dependencies, no assets, no network: double-click `tower-defense.html` to play offline. All graphics are canvas-drawn, all sound is synthesized via WebAudio (oscillators + filtered white noise).

`index.html` is a tiny `<meta http-equiv="refresh">` **redirect** to `tower-defense.html` (added in the public-release commit) so the GitHub Pages **root URL** lands on the game; it carries no game code. The Pages deploy workflow (`.github/workflows/`) copies `index.html` + `tower-defense.html` + `tower-defense.css` + `cd-*.js` + the **PWA trio** (`manifest.webmanifest` + `sw.js` + `icon.svg`, v1.30.0) into `_site` and publishes them as a **static** deploy (no build step) on every push to `master`. Don't break it.

**PWA / installable (v1.30.0):** the **hosted** (http/https) game is an installable, offline-cacheable PWA. `manifest.webmanifest` (name/short_name/`start_url:./tower-defense.html`/`display:standalone`/theme+bg `#0d1117`/maskable `icon.svg`), `icon.svg` (gold-⚡ maskable app icon matching the inline favicon), and `sw.js` (a service worker: precache the app shell in a versioned cache `circuit-defense-<GAME_VERSION>` (e.g. `circuit-defense-v1.32.1`), cache-first fetch with a same-origin runtime cache + offline fallback to `tower-defense.html`; **bump the `CACHE` const to the new `GAME_VERSION` every release** so `activate` evicts the stale shell — else hosted/installed PWA users keep an old cached game offline. **Test `[49]` asserts `sw.js`'s cache version === `GAME_VERSION`** (added v1.32.1 after the const silently drifted v1.30.0→v1.32.0), so a forgotten bump now fails the suite). The HTML head links the manifest + apple-touch-icon + iOS web-app meta tags. **SW registration lives at the end of `cd-render.js`'s startup block, guarded to `location.protocol` http/https only** — it can't register (or throw) on `file://`, so double-click play **and the headless harness (also `file://`) are byte-identical/unaffected**. NOT a module, no build step, no runtime network dep, **no localStorage/save change** (the SW uses the separate Cache API). Test group `[49]`.

**JS load order (classic scripts share one global scope, so order = dependency order — do not reorder, and keep the startup-init block last in `cd-render.js`):**

1. `cd-core.js` — canvas refs (`cv`/`ctx`/`W`/`H`), `GAME_VERSION`/`CHANGELOG_ENTRIES` + What's-New panel, audio (`tone`/`noise`/`SFX`)
2. `cd-maps.js` — `MAPS`/`genPathWith`/`buildPath`/`pointAt`, mayhem `WAVE_MODS`, stars, difficulty
3. `cd-defs.js` — `TALENTS`/`loadMeta`/meta, `TOWER_TYPES`/`MODES`/`SPECS`, `ABILITIES`, `PERKS`/`REPEATABLE`/`openDraft`
4. `cd-state.js` — run state globals, kill-combo vars, `saveRun`/`loadRun`/`resetState`
5. `cd-game.js` — start screen/`beginGame`, enemies/`startWave`, shop, upgrade panel, `eff*` stats, input listeners
6. `cd-update.js` — `update()`, `ACHIEVEMENTS`/`grantAchievements`, `recordBest`/`applyRecordFlourish`, `endGame`/`winGame`
7. `cd-render.js` — `draw()`, the rAF `loop`, and the startup init calls (`loadMeta`/`buildPath`/`renderStartScreen`/`resetState`/`setActiveUI`/`initWhatsNew`/`requestAnimationFrame(loop)`)

When adding code, drop it in the file whose domain it matches; new globals are visible everywhere as long as the file defining them loads before any top-level code that runs them (function bodies are fine anywhere — they resolve at call time).

## Running / testing

- Dev server: `.claude/launch.json` defines a `game` config (`python -m http.server 8123`). Use the preview tools and navigate to `/tower-defense.html`. (Note: `preview_screenshot` tends to time out because the rAF render loop keeps the page busy — verify state with `preview_eval`/`preview_snapshot` instead.)
- **Automated test harness lives in `tests/`** (Node + Playwright, dev-only — the shipped game is just the three raw files, no build step). Run with `cd tests && npm install && npx playwright install chromium && npm test`; exit code 0 = green. It drives the real `tower-defense.html` headlessly via an injected in-page driver: `__cdGodTowers(n)` pushes near-invincible towers, `__cdDrive({maxWave, cap})` runs the sim with `update(1/60)` to a wave boundary or game end, auto-picking drafts. Use the `check(...)` helper for assertions; add one whenever you add behavior. Before committing, spawn a subagent to run `npm test`.
- **The game loop runs on `requestAnimationFrame`, so it pauses when the tab is hidden.** Headless testing through preview_eval must drive the simulation manually: call `update(1/60)` in a loop instead of waiting wall-clock time.
- Standard test recipe (via preview_eval):
  1. `beginGame()` (set `gameMode`/`mapKey`/`diffKey` first), give `gold`, push tower objects directly into `towers`
  2. `startWave()`, then `while (!done) update(1/60)`
  3. If `draftOpen`, click a card: `document.getElementById('draftCards').children[0].click()`
  4. Check `preview_console_logs` for errors afterwards
- Synthetic canvas clicks must set real `clientX/clientY` relative to `cv.getBoundingClientRect()` — the click handler recomputes coords from the event.
- **Always clean up test data afterwards**: remove localStorage keys `cd_save`, `cd_meta`, `cd_campaign`, `cd_best_easy/normal/hard`, reset `meta = {chips:0, talents:{}}; loadMeta()`, and call `backToMenu()` so the user starts fresh.

## Architecture (code split across the seven `cd-*.js` files above)

Rough section order in the file:

- **Audio**: `tone()` (oscillator), `noise()` (filtered white noise buffer), `SFX` object — every tower type has a distinct sound (tesla = lightning crackle, cannon = sub-bass boom, etc.). Both `tone()`/`noise()` route through a single master `GainNode` (`masterGain()`, v1.13.2) instead of `a.destination`, so the Settings **Volume** slider (`setVolume()`, persisted `cd_vol` 0..1, default 0.7) scales everything; `muted` (`cd_mute`) still hard-skips all sound independently.
- **Maps & paths**: `MAPS` (classic/spiral/serpent/mayhem), `genPathWith(rnd)` generates random axis-aligned serpentine paths, `buildPath()` fills `waypoints/segs/pathLen`, `pointAt(dist)`, `distToPath(x,y)`
- **Map themes** (v1.13.8, owner FEEDBACK "all the maps look the same"): `THEMES` palette table + `pickMapTheme()`/`mapPalette()` in `cd-maps.js`, run-only `mapTheme` key. A palette (`bgIn/bgOut/star/grid/pDark/pMid/pLite/glow/dash`) feeds `draw()`'s background gradient, starfield, grid, and all path bevel/glow/dash layers (previously hardcoded blue). Each named quick-map has a **fixed** identity (`MAP_THEME`: classic→`circuit` blue, spiral→`verdant` green, serpent→`ember` amber); **campaign** rolls a tame palette per attempt from `CAMPAIGN_THEMES` (always a valid static `THEMES` key, never chaos); **mayhem** uses the **animated `chaos` palette** — `mapPalette()` computes HSL hues that sweep over `performance.now()` for a "world on fire" look, collapsing to a static fiery palette under `reduceMotion()`. `mapTheme = pickMapTheme()` is set in `resetState()`; the start-screen map buttons set it too for a live preview. **Save-safe:** the resolved key is stored in `cd_save` (`mapTheme`) and restored in `loadRun()` (so a resumed campaign keeps its random colours); old saves lack the field and fall back to the `pickMapTheme()` default. Purely cosmetic — no gameplay/economy impact. Note `resetState()` already regenerates `MAPS.mayhem.pts` every `beginGame()`, so mayhem's path is random per run independent of theming. Test group `[28]`.
- **Daily Challenge** (v1.28.0, ROADMAP "Daily challenge seed"): a 🗓 start-screen button (`beginDaily()`, cd-game.js) launches a **date-seeded** run. Modelled as a quick-mode Mayhem run (`gameMode='quick'`, `mapKey='mayhem'`, so `isMayhem()` stays true → chaos theme + wave mods) with a run-only `daily=true` flag (declared in cd-maps.js alongside `dailyDateKey`/`dailySeed`/`dailyMods`) that swaps the random bits for a **deterministic seeded stream** and **fixes the map** (no every-5-waves shift). `setupDaily(dateStr?)` (cd-maps.js): `dailySeed = dailySeedFrom(dailyDateString())` (FNV-1a over local `YYYYMMDD`), one `mulberry32(seed)` stream resolves — in fixed order — `diffKey` (normal/hard only; never easy), `MAPS.mayhem.pts = genPathWith(rnd)` (the fixed daily path), and `dailyMods[1..30]` (per-wave mod-id-or-null at the same 78% rate). `rollWaveMod()` reads `dailyMods[wave]` when `daily`. Per-day best wave under additive key **`cd_daily_<YYYYMMDD>`** — `bestKey()` (cd-state.js) routes to it when `daily`, so HUD `best`, `recordBest()` writes, and the "new record" flourish all use it; `recordBest()` (cd-update.js) captures the prior value before the write for the flourish, and **does NOT touch the per-map quick keys**. The Daily button + Records panel surface today's best. **One-off / not resumable / never touches the normal save:** `beginDaily()` skips `clearRun()`, `saveRun()` bails when `daily`, `endGame()` skips `clearRun()` when `daily`, and `loadRun()`/`beginGame()`/`backToMenu()` clear the flag. `startWave()` and the "world will shift" render hint both gate `shiftWorld` on `!daily`. `resetState()` keeps the seeded path (`isMayhem() && !daily` guard) and labels the run `🗓 Daily <date>`. Offline-safe (local date, no network). Test group `[47]`. **Daily streak (v1.31.0):** consecutive calendar days the player *finishes* a daily run (any outcome). One additive key `cd_daily_streak = {count, last:'YYYYMMDD'}`. `dailyDayBefore(key)` (cd-maps.js) computes the prior day; `recordDailyStreak()` (cd-update.js, called from `endGame`/`winGame` when `daily`) is one-per-day extend-or-reset; `currentDailyStreak()` (read-only) returns the standing count but lapses to 0 if the last finish is older than yesterday. Surfaced as `🔥{n}d` on the Daily button (`renderStartScreen`, ≥2) and a `🔥 Daily streak` row in `renderBests()`. Old saves default to 0; `resetAllData()` clears it via the cd_-prefix sweep. No gameplay/economy/schema impact. Test group `[50]`. **Daily button preview (v1.47.0):** `dailyPreview(dateStr?)` (cd-maps.js, beside `MOD_BY_ID = {id→WAVE_MODS entry}`) is a **read-only** mirror of `setupDaily()`'s seeded stream — it consumes the rng in the *exact same order* (difficulty roll → `genPathWith(rnd)` to advance the stream → the 30-wave `rnd()<0.78 ? pick : null` schedule, **including the short-circuit** so the 2nd `rnd()` is only consumed on a passing roll) but writes **nothing** to globals, returning `{diff, modIds:[…distinct, in first-appearance order]}`. `renderStartScreen()` (cd-game.js) uses it to show today's difficulty + up to 5 mod **icons** on the 🗓 Daily button (full mod **names** in the `title` tooltip). - **Modes**: `gameMode` = `'quick'` | `'campaign'`; `campLevel` 1–40, `CAMPAIGN_LEVELS = 40`. Campaign maps are **freshly random each attempt** (deliberate design choice). `victoryWave()` = `14 + campLevel` for campaign (L1=15 … L40=54), 30 for quick. Campaign enemy HP scales `1 + (campLevel-1)*0.04`. Base per-enemy HP (`enemyTemplate`) = `(18 + w*7 + 1.25*w^1.9) * 1.80 * d.hp * campScale` — tuned iteratively per recurring "too easy" FEEDBACK: the global multiplier `1.2` → `1.44` (v1.9.2) → `1.80` (v1.10.0) were *uniform* up-shifts; then v1.13.3 put a `1.25` coefficient on the **superlinear `w^1.9` term** to *steepen the curve* (owner: early waves hard but the ramp plateaus). Scaling only that term means the boost grows with wave (~+7% at w5, +12% at w10, +18% at w30) and **asymptotes toward +25% without ever exceeding it** — staying inside the ≤25%/number/run guardrail at every wave incl. deep endless (an exponent bump would be unbounded). **⚠ The norm-HP curve is now at its design ceiling: the `1.25` coeff already sits at the +25%-vs-(coeff-1.0)-baseline asymptote that test `[16]` enforces at *every* wave, so it can't be pushed further (toward the old ROADMAP "1.55") without breaking that invariant + the guardrail — late difficulty is raised via the boss/economy levers instead, see ROADMAP.** Bosses (every 5th wave) and tanks scale off the same template: **boss HP = `template × (14 + w*0.6)`** (the slope was steepened `0.5 → 0.6` in v1.24.4 to harden the late-game plateau — +3% at w5 → +10% at w30, bounded to a +20% asymptote by the `+14` constant, inside the ≤25%/run cap; the boss mult is NOT covered by test `[16]`'s norm-HP invariant, which is why it's the open late lever); boss armor `w*0.4`, bounty `template_bounty × 12`. Test group `[44]`. **Boss archetypes (v1.25.0, ROADMAP "Boss variety" + serves the recurring "too easy" FEEDBACK off the HP axis):** from **wave 20+** each boss also carries a **mechanic** assigned in `buildWave` by `BOSS_ARCHETYPES[(w/5−4)%5]` (`BOSS_ARCHETYPES = ['regen','summoner','bulwark','enrager','teleporter']` in cd-game.js — **enrager added v1.34.0 (cycle 3→4), teleporter added v1.40.0 (4→5)**), so deep bosses cycle regen→summoner→bulwark→enrager→teleporter (w20/25/30/35/40; w45 wraps to regen).
