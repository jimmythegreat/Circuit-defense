# Circuit Defense

A browser tower defense game. **The game is `tower-defense.html` (markup) + `tower-defense.css` (styles) + seven domain-split JS files (v1.8.2).** Everything is wired with classic `<link rel="stylesheet">` and `<script src>` tags — **NEVER ES modules** (`type="module"` breaks `file://`). No build step, no dependencies, no assets, no network: double-click `tower-defense.html` to play offline. All graphics are canvas-drawn, all sound is synthesized via WebAudio (oscillators + filtered white noise).

`index.html` is a tiny `<meta http-equiv="refresh">` **redirect** to `tower-defense.html` (added in the public-release commit) so the GitHub Pages **root URL** lands on the game; it carries no game code. The Pages deploy workflow (`.github/workflows/`) copies `index.html` + `tower-defense.html` + `tower-defense.css` + `cd-*.js` into `_site` and publishes them as a **static** deploy (no build step) on every push to `master`. Don't break it.

**JS load order (classic scripts share one global scope, so order = dependency order — do not reorder, and keep the startup-init block last in `cd-render.js`):**

1. `cd-core.js` — canvas refs (`cv`/`ctx`/`W`/`H`), `GAME_VERSION`/`CHANGELOG_ENTRIES` + What's-New panel, audio (`tone`/`noise`/`SFX`)
2. `cd-maps.js` — `MAPS`/`genPathWith`/`buildPath`/`pointAt`, mayhem `WAVE_MODS`, stars, difficulty
3. `cd-defs.js` — `TALENTS`/`loadMeta`/meta, `TOWER_TYPES`/`MODES`/`SPECS`, `ABILITIES`, `PERKS`/`REPEATABLE`/`openDraft`
4. `cd-state.js` — run state globals, kill-combo vars, `saveRun`/`loadRun`/`resetState`
5. `cd-game.js` — start screen/`beginGame`, enemies/`startWave`, shop, upgrade panel, `eff*` stats, input listeners
6. `cd-update.js` — `update()`, `ACHIEVEMENTS`/`grantAchievements`, `recordBest`/`applyRecordFlourish`, `endGame`/`winGame`
7. `cd-render.js` — `draw()`, the rAF `loop`, and the startup init calls (`loadMeta`/`buildPath`/`renderStartScreen`/`resetState`/`setActiveUI`/`initWhatsNew`/`requestAnimationFrame(loop)`)

When adding code, drop it in the file whose domain it matches; new globals are visible everywhere as long as the file defining them loads before any top-level code that runs them (function bodies are fine anywhere — they resolve at call time).

## Running / testing

- Dev server: `.claude/launch.json` defines a `game` config (`python -m http.server 8123`). Use the preview tools and navigate to `/tower-defense.html`. (Note: `preview_screenshot` tends to time out because the rAF render loop keeps the page busy — verify state with `preview_eval`/`preview_snapshot` instead.)
- **Automated test harness lives in `tests/`** (Node + Playwright, dev-only — the shipped game is just the three raw files, no build step). Run with `cd tests && npm install && npx playwright install chromium && npm test`; exit code 0 = green. It drives the real `tower-defense.html` headlessly via an injected in-page driver: `__cdGodTowers(n)` pushes near-invincible towers, `__cdDrive({maxWave, cap})` runs the sim with `update(1/60)` to a wave boundary or game end, auto-picking drafts. Use the `check(...)` helper for assertions; add one whenever you add behavior. Before committing, spawn a subagent to run `npm test`.
- **The game loop runs on `requestAnimationFrame`, so it pauses when the tab is hidden.** Headless testing through preview_eval must drive the simulation manually: call `update(1/60)` in a loop instead of waiting wall-clock time.
- Standard test recipe (via preview_eval):
  1. `beginGame()` (set `gameMode`/`mapKey`/`diffKey` first), give `gold`, push tower objects directly into `towers`
  2. `startWave()`, then `while (!done) update(1/60)`
  3. If `draftOpen`, click a card: `document.getElementById('draftCards').children[0].click()`
  4. Check `preview_console_logs` for errors afterwards
- Synthetic canvas clicks must set real `clientX/clientY` relative to `cv.getBoundingClientRect()` — the click handler recomputes coords from the event.
- **Always clean up test data afterwards**: remove localStorage keys `cd_save`, `cd_meta`, `cd_campaign`, `cd_best_easy/normal/hard`, reset `meta = {chips:0, talents:{}}; loadMeta()`, and call `backToMenu()` so the user starts fresh.

## Architecture (code split across the seven `cd-*.js` files above)

Rough section order in the file:

- **Audio**: `tone()` (oscillator), `noise()` (filtered white noise buffer), `SFX` object — every tower type has a distinct sound (tesla = lightning crackle, cannon = sub-bass boom, etc.). Both `tone()`/`noise()` route through a single master `GainNode` (`masterGain()`, v1.13.2) instead of `a.destination`, so the Settings **Volume** slider (`setVolume()`, persisted `cd_vol` 0..1, default 0.7) scales everything; `muted` (`cd_mute`) still hard-skips all sound independently.
- **Maps & paths**: `MAPS` (classic/spiral/serpent/mayhem), `genPathWith(rnd)` generates random axis-aligned serpentine paths, `buildPath()` fills `waypoints/segs/pathLen`, `pointAt(dist)`, `distToPath(x,y)`
- **Map themes** (v1.13.8, owner FEEDBACK "all the maps look the same"): `THEMES` palette table + `pickMapTheme()`/`mapPalette()` in `cd-maps.js`, run-only `mapTheme` key. A palette (`bgIn/bgOut/star/grid/pDark/pMid/pLite/glow/dash`) feeds `draw()`'s background gradient, starfield, grid, and all path bevel/glow/dash layers (previously hardcoded blue). Each named quick-map has a **fixed** identity (`MAP_THEME`: classic→`circuit` blue, spiral→`verdant` green, serpent→`ember` amber); **campaign** rolls a tame palette per attempt from `CAMPAIGN_THEMES` (always a valid static `THEMES` key, never chaos); **mayhem** uses the **animated `chaos` palette** — `mapPalette()` computes HSL hues that sweep over `performance.now()` for a "world on fire" look, collapsing to a static fiery palette under `reduceMotion()`. `mapTheme = pickMapTheme()` is set in `resetState()`; the start-screen map buttons set it too for a live preview. **Save-safe:** the resolved key is stored in `cd_save` (`mapTheme`) and restored in `loadRun()` (so a resumed campaign keeps its random colours); old saves lack the field and fall back to the `pickMapTheme()` default. Purely cosmetic — no gameplay/economy impact. Note `resetState()` already regenerates `MAPS.mayhem.pts` every `beginGame()`, so mayhem's path is random per run independent of theming. Test group `[28]`.
- **Modes**: `gameMode` = `'quick'` | `'campaign'`; `campLevel` 1–40, `CAMPAIGN_LEVELS = 40`. Campaign maps are **freshly random each attempt** (deliberate design choice). `victoryWave()` = `14 + campLevel` for campaign (L1=15 … L40=54), 30 for quick. Campaign enemy HP scales `1 + (campLevel-1)*0.04`. Base per-enemy HP (`enemyTemplate`) = `(18 + w*7 + 1.25*w^1.9) * 1.80 * d.hp * campScale` — tuned iteratively per recurring "too easy" FEEDBACK: the global multiplier `1.2` → `1.44` (v1.9.2) → `1.80` (v1.10.0) were *uniform* up-shifts; then v1.13.3 put a `1.25` coefficient on the **superlinear `w^1.9` term** to *steepen the curve* (owner: early waves hard but the ramp plateaus). Scaling only that term means the boost grows with wave (~+7% at w5, +12% at w10, +18% at w30) and **asymptotes toward +25% without ever exceeding it** — staying inside the ≤25%/number/run guardrail at every wave incl. deep endless (an exponent bump would be unbounded). Bosses/tanks scale off the same template. Progress = localStorage `cd_campaign` (highest completed level). Mayhem (quick-mode map) shifts the path + relocates towers every 5 waves and rolls per-wave `WAVE_MODS`.
- **Meta / talents**: `TALENTS` (21 talents in CORE + TOWER MASTERY sections — one `mastery_<type>` per tower incl. `mastery_mortar` added v1.23.0), `meta = {chips, talents, achievements, stats}` in localStorage `cd_meta`. Chips earned per run via `chipsForRun()`. `maxTowerLevel()` = 5 + overdrive rank. `loadMeta()` additively migrates older saves (defaults `achievements`/`stats` when absent — never break this).
- **Achievements** (v1.5.0, 9 badges as of v1.8.0): `ACHIEVEMENTS` array + `ACH_BY_ID`. `meta.achievements` is an id→true map; `meta.stats = {dmg, runs, bestCombo}` are lifetime counters (`bestCombo` added v1.8.0, defaulted to 0 in `loadMeta()`). `grantAchievements(won)` is called once per finished run from `endGame(false)`/`winGame(true)` — it tallies `stats` (including `meta.stats.bestCombo = max(bestCombo, comboBest)` from the run's peak streak), evaluates each badge's condition, and returns the newly-unlocked list, which `achLine()` appends to the overlay text. Per-run `livesLostThisRun` flag (set wherever a life leaks; forced `true` in `loadRun()` so resumed runs can't earn **Flawless**). The **Combo Master** badge (`combo30`, v1.8.0) is granted when the run's `comboBest >= 30`. Start-screen 🏅 button → `openAchievements()`/`renderAchievements()` panel (`#achPanel`, mirrors talents); `#achBtn` shows `done/total` (updated in `renderStartScreen`). Adding a badge = push to `ACHIEVEMENTS` + a condition line in `grantAchievements`.
- **Records / personal bests** (v1.6.0): start-screen 🏆 button → `openBests()`/`renderBests()` panel (`#bestPanel`, mirrors achievements). Shows a `.bestTbl` matrix of highest wave per quick-mode **map × difficulty**, an `★ Any map` row (from the legacy per-diff bests), and a `.bestStats` footer (campaign level, lifetime dmg, **best combo** [v1.8.0], runs, chips). Best-wave is recorded by `recordBest()` (called from `endGame`/`winGame`, replacing the old inline `if (wave > best)` lines): it updates the legacy `cd_best_<diff>` key (via `bestKey()`) **and**, in quick mode only, a new additive `cd_best_<map>_<diff>` key. Campaign runs never write per-map keys (maps are random per attempt). All keys read with `|| 0` fallback — old saves unaffected. Map keys (classic/spiral/serpent/mayhem) can't collide with diff keys (easy/normal/hard). **New-record flourish (v1.6.1):** `recordBest()` returns a `{prev, now}` event when a quick-mode run beats its per-map cell (only when `prev > 0` — first-ever entries record silently and return `null`); `endGame()`/`winGame()` pass it to `applyRecordFlourish(rec)`, which shows the golden `#ovRecord` banner (toggled by an `.record` class on `#overlay`, CSS pop/pulse animations), plays `SFX.record()` (delayed ~450ms), and adds shake + a particle burst. `applyRecordFlourish(null)` clears the banner, so every overlay show resets it (no stale banner). DOM/audio only — no save impact; campaign never fires it.
- **End-of-run score & overlay** (v1.16.0, owner FEEDBACK "a scoring system … kill time, remaining gold, using fewer towers" + "victory screen … overwhelming, restyle it"): `computeScore()` (cd-update.js) returns `{score, parts, diffMult, effMult, nt}` — `score = (wave*100 + kills*5 + lives*120 + floor(gold) + comboBest*25 + campLevel*200[campaign only] + 2500[if victory]) * DIFFS[diffKey].chipMult * effMult`, where `effMult = towers.length ? 1 + max(0,10-towers.length)*0.03 : 1` (≤10 towers → up to +30%, the "fewer towers" lever). `scoreGrade()` returns `{g,c}` (letter + colour) from `wave/victoryWave()`: **S** if `victory && !livesLostThisRun`, **A** any win, **B/C/D** at ≥.75/.5/.25, else **F**. `renderEndScreen(won, earned, newAch)` (called from `endGame`/`winGame` in place of the old `ovText` blob) fills `#ovScore` (grade badge `.ovGrade` + number + best), a concise `#ovText` headline, and `#ovDetails` (a `.scoreGrid` of 6 cells + `.ovSection` MVP/perks/`.ach` lines), and adds `.scored` to `#overlay` to reveal the hero. All-time best persists in the additive `cd_bestscore` key (read `|| 0`, `try/catch` write) — shows `Best: N` or a ★ `.newbest` flag when beaten. **Purely cosmetic + one additive key** — `computeScore` only *reads* run state, no economy/balance/save-schema impact; campaign-complete title override still wins (renderEndScreen doesn't touch `#ovTitle`). Test group [31]. The CSS lives in the `#ovScore`/`#ovDetails`/`.scoreGrid`/`.ovSection` block in tower-defense.css.
- **Version / What's New**: `GAME_VERSION` constant (semver-ish, bumped every routine run) + `CHANGELOG_ENTRIES` array (**FULL history, newest first** — mirrors CHANGELOG.md) near the top of the script. **⚠️ Owner preference (v1.13.5): the What's New panel must show EVERY past update — do NOT trim `CHANGELOG_ENTRIES` to a recent-N window.** (v1.13.4 trimmed it to the last 10; the owner reverted that the same day and asked for the complete list — the panel scrolls, so length is fine. Keep prepending new entries and never prune old ones unless the owner asks in writing.) **Layout (as of v1.4.1):** the whole game lives in a `#gameCol` column (h1, `#hud`, `#stage`→`#gameWrap`/canvas, `#shop`, `#controls`, `.hint`), and the `#whatsnew` `<aside>` is its **sibling** inside an outer `#appRow` flex row (`align-items:flex-start; flex-wrap:wrap`). So the panel floats beside the **entire game** — opening it shifts the whole column together (not just the canvas, which was the v1.4.0 complaint). The panel is **open by default**, top-aligned with the game column, flush against its right edge (`#appRow` gap 0, `border-left:none`, `border-radius:0 8px 8px 0`, `align-self:stretch`), **height-capped to the game** (v1.5.2): a `syncWhatsNewHeight()` helper sets the panel's inline `max-height` to `#gameCol.offsetHeight` (called from `openWhatsNew()` and on `window` `resize`), so on tall viewports it never overhangs the bottom of the board — overflowing entries scroll internally via `#wnList` instead. (The static CSS `max-height:88vh` stays as a no-JS fallback but the inline cap wins; the row is `align-items:flex-start`, so `#gameCol`'s height is independent of the panel and there's no feedback loop.) It wraps below `#gameCol` on narrow viewports. `initWhatsNew()` (called at startup) opens it unless the player closed it before — `closeWhatsNew()` persists `cd_wnclosed='1'`, `openWhatsNew()` clears it. The start screen has a `✨ What's New` button + clickable `#verTag` (version, set in `renderStartScreen`) to reopen, and the panel has a `✕` (`#wnClose`) to hide it. When you ship a run, bump `GAME_VERSION` and prepend a `CHANGELOG_ENTRIES` entry. **Each entry carries `v`/`date`/`body` plus an optional `time` field (v1.13.7, owner FEEDBACK "list the time as well as the date") — always include `time` (local, e.g. `'18:51 EDT'`) on new entries; `renderWnList()` shows `date · time` when present and falls back to date-only for older timeless entries.**
- **Towers**: `TOWER_TYPES` (gun/sniper/frost/cannon/tesla/poison/**mortar**/buff — 8 as of v1.23.0), `SPECS` (2 specializations per type, chosen free at level 5+), targeting `MODES` (first/last/strong/close). The shop, hotkeys (`1`..`8`), placement, spec-choice and save/restore all **auto-generate** from `TYPE_KEYS`/`TOWER_TYPES`, so a new tower is mostly additive: a `TOWER_TYPES` entry + `SPECS` pair + (optionally) a `mastery_<type>` talent, plus a `hitEnemy` branch + projectile-render handling if it uses a new `proj` kind. **Mortar (`proj:'mortar'`, v1.23.0, ROADMAP "new tower"):** an 8th tower — **long-range (225), slow (rate 2.0), lobbed AoE that always ignores armor** (back-line siege/anti-armor, distinct from the Cannon's mid-range armor-respecting splash). Deliberately a **side-grade, not power creep** (re: "too easy" feedback): dmg 28 → ~14 DPS, *below* Cannon's ~22, so it only out-performs vs *armored* targets where it bypasses the reduction; the slow reload is the lever. Firing (`cd-update.js`) forces `ignoreArmor` and a slow lob speed (200); a new `hitEnemy` branch does an armor-ignoring blast (radius 46 × Saturation 1.55 × `splashMult`). `effDmg` handles the **Demolisher** spec (+35%); **Saturation** (+55% radius) lives in `hitEnemy` (mirrors cannon's cluster/mega split). `SFX.mortar()` is the launch sound; impact reuses `SFX.bomb()`. Render (`cd-render.js`) draws `bomb`|`mortar` shells with the chunkier trail/orb. Save-safe (additive; `loadRun` rebuilds it generically). Test group [42]. **Spec rework (v1.10.0, owner FEEDBACK "dominated specs"):** to keep each L5 choice a real axis (not a trap where one option has zero damage upside), booster **Network** now adds **+10% aura power** on top of +50% range (wired in `effBuffPower`), and cannon **Mega Blast** now adds **+15% damage** alongside its blast-radius boost (`effDmg` + the `1.6` radius mult in `hitEnemy`). Overclock (+20% power) / Cluster (+50% dmg) remain the concentrated-power picks. Frost **Shatter** was *nerfed* the same run (`×6 → ×4.5`, −25%): paired with Frost Mastery + a L6 booster it was a run-warping solo carry (owner mid-run sim: frost doing 764 dmg). **Booster aura taper (v1.16.2, owner FEEDBACK "single gunner + maxed booster carries campaign-6-hard"):** `buffPower` per-level scaling cut **+0.1 → +0.08** (base 0.25 at L1 unchanged → **+65% at L6**, was +75%) — synced across all three sites (`upgradeTower` in cd-game.js, `ascendTowers` in cd-defs.js, and the `loadRun` rebuild `0.25 + 0.08*(lvl-1)` in cd-state.js, so resumed boosters keep identical power). Sim: buffed gunner 14.0→13.2 at L6 (−5.7%). Test group [33]. Frost Mastery's per-rank dmg (or a booster soft-cap) remain the next snowball levers on ROADMAP. **Booster aura *range* cut (v1.20.1, FEEDBACK "reduce booster range by 50% base … makes network better"):** `TOWER_TYPES.buff.range` `90 → 68` (−24.4% — first ≤25%/run slice of the owner's −50%; ~−43% coverage area), feeding `effBuffRange` (`= t.range × (network?1.5:1) × (1+0.02·mastery_buff)`). One booster now reaches fewer towers (chips at the solo-carry from the coverage side) and the +50% Network spec matters more off the smaller base. Save-safe — fresh placement (cd-game.js) and `loadRun` rebuild (cd-state.js) both read `def.range`. Test group [39]. Remaining reduction toward 45 queued in FEEDBACK.
- **Poison** (`proj:'poison'`, base dmg **7** as of v1.10.0): on hit, `hitEnemy()` applies a DoT `{dps: p.dmg*2.6, t: perkState.poisonDur}` (Virulent spec ×2 dps, Lingering ×2 dur, Virology perk +2s) **and corrodes the target's armor by −3 (floored at 0)** — a hard counter to shield/armored/boss enemies (owner FEEDBACK "poison should reduce enemy defense + more DoT", v1.10.0). DoT ticks via `damage(e, dps*dt, src, silent)` each frame (per-tick dmg is tiny so armor barely blunts it; the up-front impact hit is what armor corrosion helps). All run-only/live-computed — no save impact.
- **Abilities**: `ABILITIES` (meteor Q / freeze W / gold rush E), `abilityCd`, meteor is click-to-target via `armedAbility`
- **Run perks**: `PERKS` with rarity tiers (common/rare/legendary), drafted 1-of-3 every 5 waves (`openDraft`, weighted by `rollRarity()` ≈ 78/14/8 common/rare/legendary). Legendary = "super grades" (Diamond Core, Midas, Orbital Support, Ascension, **Last Stand**…). `perkState` (from `freshPerkState()`) holds all run-modifier numbers; `REPEATABLE` perks can be offered again. **Last Stand (`laststand`, legendary, v1.22.0):** a comeback perk — `apply` sets `perkState.lastStand=true`, and `effDmg()` then multiplies by `1 + min(0.6, 0.03·perkState.livesLost)` (+3%/life lost, cap +60% at 20 lost). `perkState.livesLost` is incremented at the **single life-leak site** in `cd-update.js` (by `dmgLives`, so a boss leak counts 5) regardless of whether the perk is held, so it's ready if drafted later. Both fields live **inside `perkState`** (added to `freshPerkState()`), so they're persisted whole by `saveRun()` and restored save-safely via `loadRun()`'s `Object.assign(freshPerkState(), s.perkState)` (old saves default to `false`/`0`). Deliberately conditional so it can't make a dominant ("too easy") run easier — a flawless run gets +0%. `upgradeKey()` already hashes `effDmg`, so the panel live-updates as lives leak. Test group `[41]`.
- **Effective stats**: `effDmg/effRate/effRange/effBuffPower/effBuffRange` combine base stats × talents × perks × specs × buffs × wave mods. **Booster auras do NOT stack — `buffMultFor` takes the max** (deliberate; UI says so).
- **Save/resume**: `saveRun()` after each wave + on quit (mid-wave quit saves `wave-1`); `loadRun()` rebuilds tower stats from base × level multipliers (dmg ×1.45, range ×1.08, rate ×0.88 per level). Random-path modes (mayhem/campaign) call `relocateTowers()` on resume.
- **Game flow**: `startWave → endWave` (interest, draft every 5 waves, auto-start timer) → `winGame`/`endGame` (chips awarded, MVP/perk lines) → `continueEndless`/`nextLevel`/`backToMenu`. `quitRun()` needs two clicks within 3s.
- **Concurrent waves (v1.12.0)**: instead of a single `spawnQueue`/`spawnTimer`, each in-flight wave is a parallel `spawners` entry (`{queue,timer}`) ticked independently in `update()` — so multiple waves spawn **simultaneously**. `startWave()` no longer bails on `waveActive`; it pushes a spawner and bumps `wave`, capped at `MAX_CONCURRENT_WAVES` (3) unsettled waves (`wave - lastSettledWave`). The Start button reads ➕ Add Wave (Space works too) until the cap. `endWave()` fires only on a full field-clear (`!spawners.length && !enemies.length && !pendingSpawns.length`) and **settles every bundled wave** `lastSettledWave+1..wave`: sums each clear bonus+interest and queues a draft per multiple-of-5 crossed via `pendingDrafts` (chained in `pickPerk`), so rushing never loses bonuses/perks; drafts always open on an empty field. Spawn state is never persisted; `saveRun()` stores `lastSettledWave` when active (mid-rush quit replays unsettled waves, no double-pay), `loadRun()` sets `lastSettledWave = wave`.
- **Idle chrome (v1.5.1)**: when no game is active, the in-game chrome (stats `#hud`, tower `#shop`, wave `#controls`, hotkey `.hint`) is dimmed (opacity .3 + grayscale + `pointer-events:none`) via an `.idle` class on `#gameCol`. `setActiveUI()` toggles it from the `started` flag and is called in `beginGame`/`loadRun` (lit), `backToMenu` (dimmed), and at startup. Chrome stays lit during the game-over overlay (still an active session); re-dims only on return to menu. Pure CSS + one helper, no save impact.
- **Kill-streak combo (v1.7.0)**: consecutive kills within `COMBO_WINDOW` (2s) build a streak. Run-only state `comboCount/comboTimer/comboBest/comboFlash` (declared by the globals, **reset in `resetState()`**, never saved). `damage()` bumps `comboCount`/refreshes `comboTimer`/sets `comboFlash=1` on every death and, at milestones (`=== 5 || (>=10 && %10===0)`), fires `SFX.combo(n)` (rising chirp, pitch by tier) + shake + an `addExplosion` burst + a `🔥 N× COMBO!` floater — **both on the center board, just below the top HUD band** (`W/2,114` / `W/2,132`) as of v1.8.4, so they clear the bottom-right combo meter, the centered "Wave clear! +bonus" round-clear floater (~y36-90), and the centered boss HP bar (~y8-32). `update()` decays `comboTimer` (→ resets `comboCount` to 0 at 0) and `comboFlash`. `draw()` renders the meter when `comboCount>=2 && comboTimer>0` in the **bottom-right corner** (`ax=W-16`, number baseline `H-26`) as of **v1.8.5** — its own empty space, clear of the whole top band (boss bar, round-clear bonus, top-right `#abilityBar`), the centered milestone pop, and the bottom-**left** wave preview: the `N×` multiplier is right-aligned to the corner with the "COMBO" label to its **left** on the same baseline, and a draining timer bar in its **own lane below** (`y=H-16`, drains toward the corner). Right-anchored so wide counts (`100×`) grow leftward, not off-edge. (History: meter was top-right in v1.7.0 → top-left v1.7.1 → bar/label relayout v1.8.3 → bottom-right v1.8.5; the v1.8.3 bar-under-label and the top-edge clipping at top-left both drove the corner move.) Color from `comboColor(n)` (green→gold→orange→red→purple at 10/20/30/50), pulse-scaled by `comboFlash`. **Purely cosmetic in-game — no gold/economy impact.** The combo state vars are run-only and never saved; the only persistent hook is at end-of-run, where `grantAchievements()` reads the run's `comboBest` peak into the lifetime `meta.stats.bestCombo` stat and grants the **Combo Master** badge at `>= 30` (v1.8.0).
- **Rendering**: pseudo-3D canvas — shaded cylinder towers, radial-gradient sphere enemies, ground shadows, path depth bevel, projectile trails, starfield, vignette, boss HP bar, screen `shake`. **Reduced motion (v1.10.0):** `reduceMotion()` (`cd-core.js`, reads `matchMedia('(prefers-reduced-motion: reduce)')` live, guarded for no-`matchMedia` envs) gates the `shake` translate at the top of `draw()` — OS "reduce motion" → no screen shake. (Particle/explosion gating + an in-game toggle are noted ROADMAP follow-ups.) ## Conventions & gotchas

- localStorage keys all prefixed `cd_`: `cd_save`, `cd_meta`, `cd_campaign`, `cd_best_<diff>`, `cd_best_<map>_<diff>` (per-map quick-mode best, v1.6.0), `cd_bestscore` (all-time best end-of-run score, v1.16.0), `cd_mute`, `cd_speed` (1×/2×/3× game-speed pref, v1.10.0), `cd_shake`/`cd_particles` (Settings panel render prefs, v1.13.0), `cd_vol` (0..1 master volume, v1.13.2), `cd_colorblind` (Settings ♿ Colorblind aid, `'1'`/absent, default off, v1.18.0), `cd_wnclosed` (What's New panel hidden by player), `cd_wnopen` (What's New explicitly opened — drives the small-screen default-collapse, v1.15.0)
- **Menu keyboard a11y** (v1.18.0→**v1.19.0**, owner ROADMAP table-stakes): start-screen panels are keyboard-navigable. `A11Y_PANELS` (`cd-core.js`) lists the panels in Esc-priority order — the four modal overlays (`talentPanel`/`achPanel`/`bestPanel`/`settingsPanel`) first, then the non-modal `whatsnew` rail (`nontrap:true`). A single document `keydown` listener: **Esc** closes the highest-priority open panel via `_topAnyPanel()` (so one Esc closes a modal, a second closes the rail; the draft modal is intentionally NOT in the list — a perk pick is required); **Tab** is trapped inside the open modal via `_topTrapPanel()` (wraps last→first / Shift+Tab first→last). `focusPanel(id)` — called from each `open*()` (`openTalents` in cd-defs, `openAchievements`/`openBests`/`openSettings` in cd-update) — records the opener (`_panelOpener`) and focuses the panel's first control; `_restoreOpenerFocus()` returns focus on Esc-close. **`panelOpen()` keys off `getComputedStyle(el).display`, NOT `offsetParent`** — the latter is `null` for the `position:fixed` panels of the ≤920px mobile layout, which would make Esc silently no-op on phones. A single `:focus-visible` CSS rule draws the blue keyboard focus ring (hidden for mouse). The four modal panels carry `role="dialog" aria-modal="true" aria-label` in the HTML. The in-game Esc (cancel ability targeting, cd-game.js) is behind a `!started` guard and panels are start-screen-only, so no conflict. **No new localStorage keys, no gameplay/economy/save impact.** Test group `[37]`.
